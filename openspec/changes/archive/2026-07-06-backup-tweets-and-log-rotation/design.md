## Context

当前 `x_monitor_nodriver.py` 脚本每次运行后：
- 推文抓取结果仅用于去重（`cache.json` 存最大推文 ID）和飞书推送，不持久化内容
- 所有日志写入项目根目录的 `x_monitor.log`，文件持续增长

## Goals / Non-Goals

**Goals:**
- 每次抓取的推文按 ID 持久化为独立 JSON 文件，存放在 `backup/` 目录，便于回溯和审计
- 日志按日期轮转，存放在 `logs/` 目录，每天一个 `YYYYMMDD.log` 文件
- 最小化对现有代码的改动，保持向后兼容

**Non-Goals:**
- 不实现推文全文归档（仅备份本次抓取结果）
- 不实现日志自动清理/归档（如删除 N 天前的日志）
- 不迁移旧的 `x_monitor.log` 到 `logs/` 目录

## Decisions

### 1. 推文备份格式：每个推文 ID 一个 JSON 文件

**选择**：`backup/<tweet_id>.json`，每个文件包含该推文完整信息。

**备选方案**：
- 一个文件存所有推文（`backup/tweets_<date>.json`）→ 单文件过大，按 ID 查询不便
- SQLite 数据库 → 过度设计，推文数据量小（每天最多 N 条）

**理由**：
- 文件命名即 ID，方便按 ID 查询
- 文件数量可控（每天最多 6 条，一年约 2000 个文件）
- 无需额外依赖

### 2. 日志轮转：按日期分割文件

**选择**：`logs/YYYYMMDD.log`，每天一个文件。

**备选方案**：
- 按文件大小轮转（如 10MB）→ 无法按日期快速定位
- 保留单个文件 + 定期清理 → 不符合用户需求

**理由**：
- 按日期分割便于排查某天的问题
- 脚本每 30 分钟运行一次，日志量不大，一天一个文件足够
- 实现简单，无需额外库

### 3. 实现方式：内联到现有代码

**选择**：直接修改 `_setup_logger()` 和 `fetch_tweets()` 方法。

**理由**：
- 改动量小，不超过 20 行
- 不涉及新模块或新依赖

## Risks / Trade-offs

| 风险 | 影响 | 缓解 |
|------|------|------|
| `backup/` 目录写入失败（磁盘满） | 推文备份丢失 | 静默跳过备份，记录警告日志 |
| `logs/` 目录写入失败 | 日志丢失 | 回退到 stderr 输出 |
| 旧日志文件仍存放在项目根目录 | 混淆 | 文档中明确说明路径变更 |
| 推文备份文件数量增长 | 目录文件过多 | 一年后约 2000 个文件，可接受 |

## Migration Plan

1. 修改 `_setup_logger()` 使用 `logs/` 目录和日期文件名
2. 修改 `fetch_tweets()` 在成功获取推文后写入 `backup/`
3. 更新 `.gitignore` 忽略 `backup/` 和 `logs/`
4. 更新 README 和 CHANGELOG
